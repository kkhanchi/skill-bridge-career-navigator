# ADR-015: Return 404 for cross-tenant access, accept 409 register leak

- **Status:** Accepted
- **Date:** 2026-04-30
- **Phase:** 3 — Authentication

## Context

Multi-tenant systems face a design question: when user B tries to
read/mutate a resource owned by user A, what do we return?

1. **403 FORBIDDEN** — honest about the outcome: "this exists but
   you can't access it." Leaks existence of resource ids across
   tenancy boundaries, which an attacker can use to enumerate.
2. **404 NOT_FOUND** — collapse "missing" and "not yours" into
   the same surface. Anti-enumeration by construction: an
   attacker guessing resource ids can't distinguish a hit from
   a miss.

And separately, on `/auth/register`:

3. **409 EMAIL_TAKEN** when a duplicate email is submitted —
   honest, but leaks which emails are registered on the service.
4. **200 "check your email"** (with no actual email sent to
   unregistered addresses) — hides existence but requires an
   email-verification infrastructure Phase 3 doesn't have.

## Decision

- **404 NOT_FOUND for all cross-tenant resource access.**
  `get_for_user(id, uid)` on profiles / analyses / roadmaps returns
  `None` whether the id doesn't exist or exists but is owned by
  another user. Handlers raise `ApiError(NOT_FOUND, ..., 404)`
  uniformly — same body as if the resource had never existed.
- **409 EMAIL_TAKEN on /auth/register.** We accept the existence
  leak as a Phase 3 tradeoff. The alternative (silent 200 on
  duplicate + email verification flow) is a substantial build-out
  that the phase doesn't scope. A future email-verification flow
  would close the leak.

## Consequences

**Easier:**

- Anti-enumeration for resources is a property of the repository
  contract, not a handler policy — `get_for_user` simply can't
  return a row the caller doesn't own. No per-handler care required.
- Test assertions are uniform: cross-tenant GET and "truly missing"
  GET both produce the same 404 envelope body. The property test
  `test_multi_tenant_isolation_prop.py` asserts exactly this.

**Harder:**

- Honest debugging: if a user mistypes an id, the 404 doesn't
  distinguish "you typed the wrong id" from "you don't own that
  resource." Acceptable — debugging trade-off for a security win.
- Register endpoint remains enumerable. Documented in the README
  (Phase 3 section) and tracked as a Phase 5+ refinement.

**Constrained:**

- The error message returned on a 404 is generic ("Profile not
  found"). It must never mention the owner, the owner's email, or
  any field from the owned resource. The multi-tenant property test
  asserts that no values from userA's resource appear in the body
  userB receives.
- Every `_for_user` method MUST do the ownership check before any
  mutation — not "mutate then roll back if not owned," because that
  would briefly make the change visible to concurrent reads on the
  same row. The SQL implementations put `user_id = user_id` in the
  WHERE of the UPDATE/DELETE itself, which is both correct and
  atomic.
